## 2.1 알람 서비스

---

### 2.2.1 즉시 트리거(실시간)
실시간 체결가 또는 분봉성 스트림을 기준으로 즉시 발생시키는 알람이다.

#### (1) 목표가 도달
- 사용자가 설정한 목표가 이상 도달 시 알람
- 상향 돌파 / 하향 이탈 모두 설정 가능하도록 확장 여지 확보

**예시 메시지**
- “삼성전자 목표가 82,000원 도달”
- “NAVER 손절가 165,000원 하회”

**검토 포인트**
- 장중 반복 발송 방지 필요
- 이미 보낸 목표가 알람은 재진입 전까지 중복 차단 필요

#### (2) 전일 종가 대비 ±5% 이상 등락
- 현재가가 전일 종가 대비 +5% 또는 -5% 이상 변동 시 알람

**예시 메시지**
- “LG에너지솔루션 +5.8% 상승 중”
- “카카오 -6.2% 하락 중”

**검토 포인트**
- 장 초반 변동성이 높으므로 첫 5~10분 예외 정책 검토 가능
- ±5% 기준은 사용자별 조정 가능하도록 파라미터화

#### (3) 거래량 급증
- 당일 누적 거래량 또는 일정 구간 거래량이 20일 평균 대비 3배 이상일 때 알람

**예시 메시지**
- “현대차 거래량이 20일 평균 대비 3.4배 증가”

**검토 포인트**
- 장중 거래량은 시간 경과에 따라 해석이 달라짐
- 단순 누적 거래량 비교보다는
    - 동일 시각 기준 평균 누적 거래량
    - 또는 종가 기준 배치 판정
      중 하나를 명확히 선택해야 함
- 1차에서 실시간 처리까지 하려면 데이터 복잡도가 높아질 수 있음

---

### 2.1.2 장 마감 후 트리거(배치)
장 종료 후 EOD 데이터를 기준으로 안정적으로 계산하는 알람이다.

#### (1) 골든크로스 / 데드크로스
** 5일 이동평균선이 20일 이동평균선을 상향 돌파하면 골든크로스
    하향 이탈하면 데드크로스

**예시 메시지**
- “POSCO홀딩스 5일선이 20일선을 상향 돌파했습니다.”

**검토 포인트**
- 종가 기준 확정값으로 계산해야 false signal 감소
- 1차에서는 종가 배치로만 운영하는 것이 현실적

#### (2) RSI 과매도 / 과매수 진입
- RSI 30 이하 진입 시 과매도
- RSI 70 이상 진입 시 과매수

**예시 메시지**
- “셀트리온 RSI 28.4로 과매도 구간 진입”

**검토 포인트**
- 단순 구간 진입 알람인지, 구간 유지 중 반복 알람을 보낼지 정책 필요
- 일반적으로는 **진입 시점 1회 발송**이 적절

#### (3) 외국인·기관 순매수 3일 연속 전환
- 외국인 순매수 3일 연속
- 기관 순매수 3일 연속
- 혹은 둘 다 동시 순매수

**예시 메시지**
- “삼성바이오로직스, 외국인 순매수 3일 연속”
- “SK하이닉스, 기관·외국인 동시 순매수 3일 지속”

**검토 포인트**
- 투자자 수급 데이터 소스의 안정성 필요
- 장 종료 후 확정 데이터 기준으로만 계산 권장

---

### 2.1.3 이벤트 트리거
가격/기술지표 외의 외부 이벤트를 감지해 알람을 발생시킨다.

#### (1) DART 공시 등록 즉시
- 보유 종목 관련 공시가 등록되면 즉시 알람

**예시 메시지**
- “한화에어로스페이스 공시 등록: 단일판매·공급계약 체결”

**검토 포인트**
- 공시 종류별 중요도 분류가 있으면 품질 상승
    - 예: 유상증자, 전환사채, 실적공시, 대규모 계약, 소송, 최대주주 변경 등
- 1차는 전체 공시 알림 + 제목 제공으로 시작하고,
  향후 중요도 랭킹 추가 가능

#### (2) 긍정 / 부정 뉴스 감지
- 보유 종목 관련 뉴스를 수집하고 감성 또는 이벤트 성격을 분류해 알람

**예시 메시지**
- “네이버 관련 긍정 뉴스 감지: 신규 AI 서비스 확장 보도”
- “카카오 관련 부정 뉴스 감지: 규제 이슈 보도”

**검토 포인트**
- 감성 분류 정확도가 핵심
- 1차에서 완전 자동 감성 분류는 오탐 가능성이 높으므로
    - 우선은 ‘뉴스 발생 알림 + 제목/언론사 제공’ 중심
    - 이후 긍·부정 분류는 보조 태그로 제공하는 전략이 현실적

# 3. 사용자 플로우 구체화

## 3.1 최초 설정 플로우

### Step 1. 포트폴리오 등록
사용자는 보유 종목과 평균 매입 단가, 보유 수량을 입력한다.

#### 입력 방식
- 수동 입력
- CSV 업로드(향후)
- 증권사 연동(향후)

#### 저장 데이터
- user_id
- account_id 또는 portfolio_id
- stock_code
- quantity
- avg_buy_price
- updated_at

### Step 2. 알람 조건 설정
사용자는 종목별 또는 전체 기본 룰을 설정한다.

#### 예시
- 목표가 설정
- 손절가 설정
- 급등락 알람 on/off
- 공시 알람 on/off
- 뉴스 알람 on/off
- 기술지표 알람 on/off

### Step 3. 대시보드 진입
시스템은 저장된 포트폴리오 기준으로
- 보유 현황
- 성과 요약
- 섹터 비중
- 밸류에이션 위치
- 공시/뉴스 피드
  를 렌더링한다.

---

## 3.2 장중 실시간 플로우

### 흐름 요약
1. 실시간 시세 수신
2. 사용자 보유 종목 여부 확인
3. 실시간 트리거 조건 계산
4. 조건 충족 시 이벤트 생성
5. 중복 발송 여부 확인
6. 알람 발송
7. 대시보드 실시간 반영

### 상세 플로우

#### 1) 시세 수집
- KIS API 또는 실시간 시세 feed에서 체결/호가/현재가 수신
- raw topic 또는 raw table에 저장

#### 2) 보유 종목 필터링
- 전체 시장 데이터 중 사용자 보유 종목만 빠르게 식별
- Redis 캐시 또는 in-memory set으로 보유 종목 집합 관리 가능

#### 3) 트리거 계산
- 목표가 도달 여부
- 전일 종가 대비 등락률 계산
- 거래량 급증 여부 계산

#### 4) 이벤트 생성
- 조건 만족 시 `alert_candidate` 이벤트 생성
- 이벤트 payload 예시
    - user_id
    - stock_code
    - trigger_type
    - trigger_value
    - occurred_at

#### 5) 중복 체크
- 같은 종목, 같은 조건, 같은 장중 구간에 이미 발송했는지 확인
- 중복 방지 키 예시
    - `user_id + stock_code + trigger_type + trade_date`

#### 6) 발송
- 푸시 알림 / 앱 내 알림 / 카카오 알림톡(향후)

#### 7) UI 반영
- 대시보드 보유 현황 현재가, 수익률, 평가금액 업데이트

### 흐름 요약
1. 일별 종가 확정 데이터 적재
2. 기술지표 계산

# 4. DE 관점 데이터 레이어 구체화

## 4.1 Raw Layer

### 대상
- KIS API 시세 원본
- DART 공시 원문
- 뉴스 원문
- 투자자 수급 원본
- 종목 메타 원본

### 목적
- 장애 발생 시 원본 재처리 가능
- 파싱 로직 변경 시 backfill 가능
- 감사/추적성 확보

### 권장 저장 형태
- append-only
- ingestion timestamp 포함
- source response 원문 JSON 보존

### 예시 테이블
- `raw_kis_quote`
- `raw_kis_trade`
- `raw_dart_disclosure`
- `raw_news_article`
- `raw_investor_flow`
- `raw_stock_master`

### 현실적 검토
- 실시간 tick 데이터를 모두 영구 저장하면 비용이 빠르게 증가할 수 있음
- 1차에서는 tick 전체 보관보다
    - 실시간 판정용 stream
    - 분 단위/일 단위 집계 영속화
      전략이 더 현실적일 수 있음

---

## 4.2 Cleansed Layer
원천 데이터의 스키마를 표준화하고, 서비스에 필요한 식별자 정합성을 맞춘다.

### 주요 처리
- 종목코드 정규화
- 종목명 / 섹터 / 시장구분 조인
- DART 공시 파싱
- 뉴스 종목 태깅
- 중복 뉴스 제거
- 결측치/이상치 보정

### 예시 테이블
- `cln_stock_price_daily`
- `cln_stock_price_intraday`
- `cln_stock_master`
- `cln_disclosure_event`
- `cln_news_event`
- `cln_investor_flow_daily`

### 현실적 검토
- 이 레이어에서 가장 어려운 것은 **뉴스 종목 태깅 품질**임
- 기업명 동음이의어, 그룹명/자회사 혼동, 일반명사 포함 이슈가 있음
- 1차에서는
    - 시가총액 상위 종목 우선
    - 정규 사전 기반 태깅
    - 오탐 방지용 화이트리스트/블랙리스트
      를 함께 운영하는 것이 현실적

---

## 4.3 Mart Layer
서비스 조회와 알람 판정에 직접 쓰이는 소비 지향 레이어다.

### 주요 역할
- 화면 조회 속도 개선
- 반복 계산 제거
- 알람 기준 사전 계산
- 분석/집계 결과 저장

### 예시 테이블

#### 포트폴리오 조회용
- `mart_portfolio_position`
    - 사용자별 종목 보유 현황
- `mart_portfolio_summary`
    - 사용자별 총 평가금액, 수익률, 벤치마크 비교
- `mart_portfolio_sector_weight`
    - 사용자별 섹터 비중
- `mart_stock_valuation_percentile`
    - 종목별 섹터 내 PER/PBR 분위수

#### 알람용
- `mart_stock_signal_daily`
    - 종목별 MA, RSI, 평균 거래량, 수급 신호
- `mart_alert_candidate`
    - 판정된 알람 후보
- `fact_alert_event`
    - 실제 발송 이력

### 현실적 검토
- 1차에서 가장 중요한 mart는 아래 3개임
    1. 사용자 보유 현황 mart
    2. 종목 기술지표 mart
    3. 이벤트 피드 mart
- 밸류에이션 분위수는 조회 성능상 **사전 계산**이 맞음
- 다만 PER/PBR 데이터 공급원이 안정적이어야 하며, 재무 업데이트 타이밍 관리가 필요함

---

## 4.4 Streaming Layer
실시간 이벤트 전파 계층이다.

### 대상 이벤트
- 실시간 시세
- 거래량 급증 판정 이벤트
- 목표가 도달 이벤트
- DART 공시 이벤트
- 뉴스 이벤트

### 구조
- Producer: 시세 수집기 / 공시 수집기 / 뉴스 수집기
- Broker: Kafka
- Consumer:
    - 알람 판정 서비스
    - 알림 발송 서비스
    - 실시간 대시보드 업데이트 서비스

### 현실적 검토
- Kafka 도입은 확장성 측면에서 적절
- 다만 1차에서 트래픽이 아주 크지 않다면,
  일부 배치성 이벤트는 Kafka 없이도 운영 가능
- 그래도 **실시간 시세 + 공시/뉴스 이벤트**는 스트리밍 구조가 향후 확장성에 유리

---

# 5. 기술 검토: 현실적인 구현 관점

## 5.1 전체 아키텍처 적합성
현재 제안한 구조는 전형적인
**Raw → Cleansed → Mart + Streaming** 구조로,
금융 데이터 서비스에 적합하다.

특히 좋은 점은 아래와 같다.
- 원본 보존과 재처리 가능성 확보
- 서비스용 사전 계산 레이어 분리
- 실시간/배치 처리 목적 분리
- 알람과 대시보드가 같은 데이터 자산을 공유 가능

다만 1차에서는 모든 것을 과도하게 실시간화하면 복잡도가 급격히 증가한다.
따라서 **무엇을 진짜 실시간으로 할지**를 좁히는 것이 중요하다.

---

## 5.2 1차에서 실시간으로 가져가도 되는 것
### 권장 실시간 범위
- 목표가 도달
- 전일 종가 대비 ±5% 이상 등락
- DART 공시 등록
- 뉴스 발생 이벤트

### 신중 검토 항목
- 거래량 20일 평균 대비 3배 급증
    - 장중 시점 정규화가 필요해 난이도 있음
- 실시간 골든크로스/데드크로스
    - 의미가 약하고 노이즈 큼
    - 종가 배치가 적절
- 실시간 RSI
    - 장중 지표 값이 흔들려 오탐 가능
    - 종가 배치 권장

즉, 1차 실시간 범위는 **가격/이벤트 중심**, 배치는 **기술지표/수급 중심**이 가장 현실적이다.

---

## 5.3 가장 어려운 데이터 품질 포인트

### (1) 뉴스 종목 태깅
가장 운영 난도가 높다.

#### 이유
- 기업명 중복
- 약칭 사용
- 지주사/계열사 혼동
- 종목명 외 일반 키워드와 충돌

#### 대응
- 종목 alias dictionary 운영
- 언론사/본문/제목 가중치 다르게 적용
- 금융 뉴스 우선 소스 사용
- 오탐 로그 수집 후 룰 개선

---

### (2) 밸류에이션 데이터 정합성
PER/PBR은 공급원마다 기준 시점과 계산 방식 차이가 있다.

#### 이슈
- 적자 기업 PER 처리
- 연결/별도 기준 차이
- 최근 분기 반영 시점 차이

#### 대응
- 한 공급원 기준으로 통일
- 음수 PER 제외 정책 명문화
- 분위수 계산 universe 정의 문서화

---

### (3) 사용자 포트폴리오 원장 정확성
평균 매입 단가가 부정확하면 전체 대시보드가 무너진다.

#### 대응
- 1차는 수동 입력 + 수정 가능 UI 제공
- 향후 거래내역 업로드 기반 자동 계산 확장

---

## 5.4 운영 관점 필수 설계

### 알람 중복 방지
동일 이벤트 반복 발송은 사용자 피로도를 급격히 높인다.

#### 최소 필요 장치
- 이벤트 해시 키
- 발송 이력 저장 테이블
- 쿨다운 시간
- 재진입 조건 정의

예:
- 목표가 돌파 알람은 최초 1회
- 다시 기준 아래로 내려갔다가 재돌파 시 재발송

---

### 장중 / 장후 데이터 분리
같은 “현재가”라도
- 장중 실시간값
- 장 마감 종가
  는 의미가 다르다.

따라서 화면과 알람 모두에서
**데이터 기준 시점(timestamp, session type)** 을 명확히 관리해야 한다.

---

### 실패 복구 / 재처리
금융 데이터 파이프라인은 외부 API 장애가 흔하다.

#### 필요 사항
- raw 재수집 가능 여부
- 배치 재실행 가능 구조
- 누락 데이터 보정 작업
- Kafka consumer 재처리 정책

---

# 7. API / 백엔드 플로우 관점 정리

## 7.1 대시보드 조회 플로우
1. 사용자가 앱에서 대시보드 진입
2. 백엔드가 사용자 보유 종목 조회
3. `mart_portfolio_dashboard` 조회
4. 종목별 보유 현황 + 성과 요약 + 섹터 비중 + 밸류에이션 + 이벤트 피드 조합
5. 응답 반환

### 포인트
- 화면 응답 속도를 위해 가급적 mart 조회 중심으로 구성
- 실시간 현재가는 별도 endpoint 또는 websocket으로 덧입히는 방식 가능

---

## 7.2 실시간 알람 플로우
1. 실시간 가격 이벤트 수신
2. 이벤트 consumer가 보유 사용자 목록 조회
3. 사용자별 룰 매칭
4. 중복 발송 여부 확인
5. 알림 발송
6. 발송 로그 저장

### 포인트
- 보유 사용자 lookup 성능 중요
- 종목 → 사용자 역인덱스 캐시가 유리

---

## 7.3 공시/뉴스 피드 플로우
1. 공시/뉴스 수집
2. 종목 태깅 및 정제
3. 이벤트 저장
4. 보유 사용자 매핑
5. 실시간 알림 또는 앱 피드 반영

### 포인트
- 공시는 정확도 우선
- 뉴스는 속도보다 태깅 품질이 더 중요

# 9. 최종 정리

## 핵심 판단
1. **실시간은 가격/공시/뉴스 중심으로 좁힌다.**
2. **기술지표와 수급은 장 마감 후 배치로 계산한다.**
3. **Mart 레이어를 중심으로 대시보드 응답 속도를 확보한다.**
4. **뉴스 종목 태깅과 알람 중복 방지가 품질의 핵심이다.**
5. **PER/PBR 분위수는 사전 계산 + 명확한 기준 정의가 필요하다.**
