# 주식 포트폴리오 기반 매수 기회 알림 서비스 — 설계 초안

> 상태: **팀 회의용 제안서** (확정 스펙 아님)
> 작성일: 2026-06-22
> 참여: 희라(서비스 백엔드) · 대홍(데이터/스트림·ML) · 해정(인프라·성능·운영)
>
> 이 문서는 희라의 투자자/개발자 관점 인터뷰를 토대로 수렴된 방향이다.

## 1. 투자자 니즈 (희라 인터뷰 결과)

### 핵심 니즈
1. **매수 타이밍 알림** — 내가 정한 조건·이벤트가 발생했을 때 매수 기회를 놓치지 않도록 알려주기.
2. **포트폴리오 종합 관리** — 여러 계좌의 보유 자산을 종합해 비중·수익률·실제 익스포저를 한눈에 파악. 구체적으로 계좌별 종합(매수금액·평가금액·수익률·비중), **ETF 풀어보기(look-through)**, **섹터별 비중**, **스냅샷 비교**.

### 투자 행태 (위 니즈의 배경)
- **투자 구성:** 장기 80~90% + 스윙 10~20%, 국내 + 미국, 주로 지수 ETF·반도체 섹터
- **매수 계기:**
  1. 관심 종목/섹터가 **펀더멘털은 탄탄한데 단기 조정**받을 때 → 저점 매수
  2. **긍정적 펀더멘털·이벤트 뉴스**가 떴을 때 → 모멘텀 매수
- **매도:** 우상향 성격의 안정적 종목을 장기 보유 → 매도는 거의 안 함 (**제품은 "매수 기회 포착"에 집중**)
- **실시간성:** 빠를수록 좋지만 *투자자에겐 필수 아님* → 빠른 반응이 의미 있는 이벤트에 한정해 의도적으로 실시간 설계
- **등록 단위:** 계층형(섹터 → 테마 → 종목). 정적 seed로 시작 → 뉴스 기반 자동 보강

## 2. 제품 정의

### 한 줄 정의
**장기 투자자를 위한 "포트폴리오 기반 매수 기회 알림" 서비스.**
보유·관심 종목과 섹터·테마를 등록하고 **어떤 이벤트에 알림을 받을지 직접 정의**해두면, 그 이벤트가 발생할 때 **내 보유/관심 기준으로 개인화된** 매수 기회를 알려준다.

### 핵심 개념
- 두 레이어: **포트폴리오 통합 관리(기반 데이터)** + **개인화 시그널 알림(그 위 레이어)**
- 알림 대상은 **보유 종목 + 관심 종목 둘 다**:
  - **보유 종목** = 포트폴리오에 입력된 실제 보유분 (추가 매수 기회)
  - **관심 종목** = 미보유지만 워치리스트에 등록 (신규 진입 기회)
- **사용자 정의 이벤트 구독:** "언제 알림을 받을지"를 시스템이 고정하지 않고 **사용자가 관심 이벤트를 정의**한다. 시스템은 그 이벤트를 감지·해석해 영향 종목을 도출하고, 사용자의 보유/관심과 매칭되면 알린다. 새로운 이벤트 유형을 더할 수 있는 **확장형 구조**.
  - *열린 질문(추후 브레인스토밍): 이벤트를 사용자가 어떤 단위·문법으로 정의하게 할지 — 프리셋 선택형 / 조건 빌더 / 자연어 등.*
- 포트폴리오가 "내 진짜 익스포저"를 알기에 시그널을 개인화할 수 있다. (ETF 풀어보기까지 가면 ETF 내부 종목 노출까지 반영해 더 정확해짐)

### 초기 제공 이벤트 유형 (예시)
확장형 구조의 출발점으로 두 가지를 기본 제공한다. (희라의 매수 계기에서 도출)
- **이벤트 유형 A — 긍정적 이벤트 뉴스:** 뉴스·공시를 이벤트로 해석 → 영향 종목 도출. *실시간 부담 낮음.*
- **이벤트 유형 B — 단기 조정 + 펀더멘털 양호:** 실시간 가격·수급 이상을 감지하고 펀더멘털 필터로 거름. *실시간성 필요.*

### 주요 기능
1. **포트폴리오 통합 관리**
   - 입력: 수동 입력 / CSV 업로드 (차후: 한투 OpenAPI 자동 동기화)
   - 종합 대시보드: 매수금액·평가금액·수익률·**종목별 비중**·**섹터별 비중**
   - (차후) **ETF 풀어보기**: ETF 보유분을 구성종목으로 분해해 실제 익스포저 산출
   - (차후) **스냅샷**: 특정 시점 포트폴리오를 저장해 과거 ↔ 현재 비교
2. **관심 종목 / 섹터 / 테마 등록** — 계층형(섹터 → 테마 → 종목), 종목 단위 집중·제외 가능
3. **이벤트 구독 규칙** — 어떤 이벤트 유형에, 어떤 종목/섹터/테마 범위로 알림 받을지 사용자가 정의
4. **개인화 시그널 알림** — 이벤트 감지 → 영향 종목 도출 → 보유/관심 매칭 → 알림(인앱 피드 + 실시간 푸시), 알림 이력 조회

### 사용 시나리오
- **A. 보유 종목 이벤트 (이벤트 유형 A)** — SK하이닉스 보유 중, "긍정 뉴스 이벤트"를 구독. "HBM4 양산 본격화" 뉴스 → 시스템이 AI 반도체 이벤트로 해석, 영향 종목에 SK하이닉스 포함 → "보유 종목 긍정 이벤트" 알림(인앱+푸시).
- **B. 관심 종목 진입 기회 (이벤트 유형 A)** — 한미반도체를 관심종목으로 등록(미보유). 관련 긍정 이벤트 발생 → "관심 종목 매수 기회" 알림.
- **C. 포트폴리오 맥락 동봉** — 알림에 "현재 네 포트폴리오의 반도체 섹터 비중 35%" 같은 맥락을 함께 제공해 의사결정 보조.
- **D. 포트폴리오 조회** — 여러 계좌 종합 → 종목별/섹터별 비중과 수익률을 한눈에.
- **E. (차후) 단기조정 매수 (이벤트 유형 B)** — 보유/관심 종목이 장중 비정상 급락(펀더멘털 양호) → 실시간 "눌림목 매수 기회" 알림.

## 3. 아키텍처 (역할 = 협업 경계)

```
거래소(체결·호가) / 뉴스·공시(DART·뉴스 API)
                    │
                    ▼  Kafka (raw topics)
 ┌──────────────────────────────────────────────────────────┐
 │  데이터팀 영역 (대홍 + 해정)                                │
 │   스피드 레이어(대홍): Flink — 캔들 집계 · 이상탐지         │
 │   배치 레이어(대홍+해정): Spark/Iceberg 메달리온            │
 │       Bronze(raw) → Silver(정규화) → Gold(피처)           │
 │       baseline·모델 재학습, Airflow/dbt 오케스트레이션     │
 │   서빙 레이어: 스피드 결과 + 배치 결과 머지                  │
 └──────────────────────────────────────────────────────────┘
                    │
        ┌───────────┴────────────┐
        ▼                        ▼
   Kafka `signals` 토픽       Gold/서빙 테이블 (조회용)
        └──────── 팀 간 계약(스키마) ────────┘
                    │
 ┌──────────────────────────────────────────────────────────┐
 │  서비스 백엔드 영역 (희라) — Spring Boot + PostgreSQL      │
 │   포트폴리오 도메인 · 구독/매칭 · 정합성/멱등성 · 실시간 서빙 │
 └──────────────────────────────────────────────────────────┘
                    │
                    ▼  (해정: k8s·클라우드·모니터링·배포)
                 [ 사용자 ]
```

- **협업 형태(분리형):** 데이터팀 ↔ 희라는 **Kafka `signals` 토픽(스키마 계약)** + **조회용 Gold/서빙 테이블** 둘로만 연결. 서로 내부 구현을 몰라도 됨.
- **희라:** 서비스 백엔드(도메인·API·정합성·실시간 서빙)
- **대홍:** 람다(스피드+배치) + 메달리온 + 이상탐지/ML
- **해정:** 인프라·Spark/Iceberg 튜닝·확장·운영

## 4. 데이터팀 ↔ 백엔드팀의 기술적 의의

각자가 이 프로젝트에서 증명·강화하려는 역량과, 그것이 어느 기능에서 드러나는지.

### 백엔드팀 (희라)
- **도메인 모델링 & API 설계:** Account · Transaction · Holding · Watchlist · Instrument(종목/ETF) · ETFConstituent · Portfolio · Snapshot · Theme · AlertRule(이벤트 구독 규칙) · Signal · Notification — 비즈니스 요건이 전제된 모델.
- **데이터 정합성 & 멱등성:**
  - 시그널 중복 수신 dedup, at-least-once 전달 → 멱등 처리(정확히 한 번 알림)
  - 수동 입력 ↔ (차후) API 데이터 충돌 해결
  - 평가금액·수익률·비중 계산의 일관성, **스냅샷의 시점 정합성**
- **실시간성(선택적 적용):** "모든 걸 실시간으로 하지 않고 필요한 곳만" — 이벤트 유형 B처럼 빠른 반응이 필요한 이벤트에 한정해 WebSocket/SSE 적용. 시니어다운 판단을 서사로.
- **(차후) 외부 API 연동:** 한투 OpenAPI OAuth 토큰 수명 관리, 동기화 멱등성.

### 데이터팀 (대홍) — 람다 + 메달리온 + 이상탐지/MLOps
- 스피드 레이어: Flink 윈도우 집계(캔들), 상태(직전 N초 baseline) 관리, exactly-once, 백프레셔·컨슈머 랙 모니터링.
- 배치 레이어: 메달리온(Bronze→Silver→Gold), Spark로 과거 데이터 재처리·baseline/모델 재학습.
- 서빙 머지: 스피드 결과(신선) + 배치 결과(정확)를 일관되게 합쳐 노출 (람다 핵심 난제).

### 데이터팀 (해정) — 성능 최적화 + 확장·운영
- Iceberg 파티셔닝·테이블 옵션 튜닝, Spark 처리 성능 개선, 쿼리 성능 최적화.
- 클라우드/k8s 인프라 구성, 확장 가능한 시스템 설계, 운영 관점 아키텍처.

> 메타 포인트: 이벤트 유형 B(단기조정 = 실시간 이상탐지 × 펀더멘털 필터)는 **대홍·해정·희라가 한 기능에서 만나는** 지점이라, 세 명의 협업을 가장 잘 증명한다.

## 5. 범위 & 로드맵

### 전체 범위 (최종 지향점)
포트폴리오 통합 관리(ETF 풀어보기·섹터비중·스냅샷) + 사용자 정의 이벤트 알림(뉴스 이벤트 + 실시간 이상탐지 등 확장형) + 한투 OpenAPI 자동연동 + 뉴스 기반 종목-테마 자동 매핑.

### MVP 범위
**원칙: 데이터 수집부터 사용자 알림까지 한 줄기를 끝에서 끝까지 관통시킨다.** 세 영역(데이터 파이프라인 · 인프라 · 서비스 백엔드)이 처음부터 각자 담당으로 동시에 움직여 협업 구조를 가장 먼저 검증하기 위함이다. 가장 만만한 한 줄기로 **"긍정적 이벤트 뉴스(이벤트 유형 A) → 영향 종목 도출 → 내 보유/관심 매칭 → 알림"**을 택한다. (실시간 부담이 없어 첫 관통에 적합)

| 담당 | MVP 범위 |
|---|---|
| 데이터팀 (대홍) | 뉴스·공시 **수집 → Kafka 적재**, **이벤트 해석**(뉴스 → 이벤트 → 영향 종목 도출), `signals` 토픽 발행. 메달리온 Bronze/Silver 최소 형태 |
| 데이터팀 (해정) | **Kafka·저장소·k8s 등 인프라 구성**, 배포 환경, 파이프라인 운영·성능 기반 |
| 서비스 백엔드 (희라) | **포트폴리오**(수동/CSV, 종목·**섹터별 비중**) + **관심 종목** 등록 + **이벤트 구독 규칙** + `signals` 소비·매칭 + **알림 서빙**(인앱 피드·푸시) |
| 공통 도메인 데이터 | 섹터 매핑(공식 업종분류), 반도체 중심 테마 seed (§6) |

> MVP는 백엔드 단독 작업이 아니다. 데이터팀이 **수집 → 이벤트화 → 시그널 발행 파이프라인 전체**를 소유하고, 희라는 그 시그널을 소비해 도메인·알림을 담당하며, 해정이 인프라를 깐다. 세 영역이 한 줄기 안에서 동시에 움직인다.

**MVP에서 제외(=의도적으로 나중):** 이벤트 유형 B(실시간 이상탐지), ETF 풀어보기, 스냅샷 비교, 한투 OpenAPI 연동, 자동 테마 매핑.

### 다음 단계 (살 붙이는 순서)
1. **ETF 풀어보기** — 실제 익스포저 정확도 ↑ (희라 도메인 + 데이터팀 ETF 구성종목 공급)
2. **이벤트 유형 B (실시간 이상탐지)** — 대홍 스피드 레이어 × 펀더멘털 필터, 희라 실시간 서빙(WebSocket/SSE)
3. **스냅샷 비교** — 포트폴리오 시계열·시점 정합성
4. **한투 OpenAPI 자동연동** — 수동/CSV에서 자동 동기화로
5. **사용자 이벤트 정의 고도화 + 뉴스 기반 종목-테마 자동 매핑** — 이벤트 정의 문법 확장, 정적 seed를 동적 점수로 보강(데이터팀 ML/배치)

## 6. 테마/종목 매핑 전략

계층 중 **섹터(Tier 1)**는 거래소 공식 업종분류(KRX/GICS)로 기계적 매핑 — 공개 데이터, 안정적. **테마(Tier 2, 예: "HBM")**는 공식 분류에 없어 별도 전략 필요.

### 왜 반도체 중심인가
1. **Dogfooding:** 반도체는 희라(실사용자)의 실제 투자 도메인 → "쓸만한 알림인가"를 직접 검증 가능.
2. **테마 경계가 명확:** HBM·파운드리·소부장 등 관련 종목이 비교적 또렷해 초기 seed 품질이 높음.
3. **두 시장 커버:** 삼성전자·SK하이닉스·한미반도체(국내) + 엔비디아·TSMC(미국) 등 종목이 풍부해 국내+미국을 한 테마로 다룸.

### 왜 (MVP는) 손으로 seed 하는가
1. **MVP 목표는 "관통":** 매핑 정확도보다 엔드투엔드 동작 검증이 우선 → 핵심 테마 5~6개 × 종목 20~30개면 충분.
2. **공식 분류의 공백:** "HBM" 같은 세부 테마는 업종분류에 없어 누군가는 정의해야 함 → 초기엔 사람이 빠르고 정확.
3. **협업 경계가 깔끔:** 자동 매핑(뉴스 co-occurrence·임베딩)은 데이터팀의 ML/배치 거리 → **정적 seed = 백엔드 도메인 / 동적 점수 = 데이터팀**으로 책임이 자연스럽게 분리. (다음 단계에서 (b) 자동 보강)
