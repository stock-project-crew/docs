## 0. 개요 & 범위

### 무엇을 만드나
1차로 두 기능을 만든다.
- **포트폴리오 종합 관리:** 여러 계좌(국내+미국)의 보유 자산을 모아 종목별·섹터별 비중과 수익/손익을 보여준다. ETF는 구성종목으로 분해(look-through)해 실제 익스포저를 계산한다.
- **매수/매도 타이밍 알림:** 사용자가 프리셋 기반 조건을 걸어두면, 시세·지표·포트폴리오 상태를 평가해 충족 시 알림을 보낸다(주문은 하지 않는다).

### 운영 모델
**개인용 도구**다 — 각자 자기 API 키로 자기 계좌만 연동해 본다. 멀티유저 서비스와의 차이는 화면(둘 다 본인 포트폴리오·알림을 봄)이 아니라 **자격증명 보관 범위**다: 서버가 불특정 다수의 증권사 키를 떠안지 않으므로 약관·보안 부담이 작다. 공개 서비스·상업화로 가면 타인 계좌 연동이 불가해져 입력 모델을 다시 설계한다(§7).

### 범위
**핵심 기능 2가지:** 포트폴리오 종합 관리, 매수/매도 타이밍 알림. **UI는 앱**, **자동 주문은 하지 않는다**(알림까지).

두 기능을 이루는 구성 요소별로 *필요 데이터 / 확보·검증 / 리스크*를 정리한다.

**포트폴리오 종합 관리**

| 구성 요소 | 필요 데이터 | 확보·검증 | 리스크 |
|---|---|---|---|
| 보유·예수금 동기화 | 잔고(보유수량·평단·평가금액·예수금) | 한투 잔고·python-kis 통합 실측 ✅ | 자동은 한투만, 타 증권사·연금은 수동 |
| 거래내역·실현손익 | 기간 체결내역 + 기업행위(분할·증자) | 위탁계좌 체결내역 실측 ✅ | 연금·비API는 수동, 과거 전체는 조회기간 한계, **기업행위 데이터 원천 미확보 시 실현손익 부정확** |
| 평가·시세 | 현재가·일봉 OHLCV | 한투 시세 API(국내·해외) — 잔고·실시간 실측, 일봉은 미실측 / yfinance(미국)은 미검증 | 비공식 시세(yfinance·토스) 신뢰성 |
| ETF 분해(look-through) | ETF 구성종목 | 미국 SPDR ✅ / 국내 원천 차단 | 국내 적용 여부는 회의 결정(§7) |
| 섹터 비중 | 종목→섹터 분류 | 미국 yfinance·국내 Naver 업종 실측 ✅ | 미·국 체계 통일 매핑 필요, 스크래핑 원천은 개인용 한정 |
| 환율(원화 환산) | 통화별 환율 | python-kis `exchange_rate` 필드 확인 | 매입시점 FX 부재(환손익 혼재), USD 실값 미검증 |
| (선택) 다증권사 통합 | 타 증권사 잔고·거래 | krsec 미검증(신생) | 1차는 한투 단독, 확장은 검증 후 |

**매수/매도 타이밍 알림**

| 구성 요소 | 필요 데이터 | 확보·검증 | 리스크 |
|---|---|---|---|
| 지표 프리셋 알림 | 일봉 OHLCV(지표 자체계산) + 포트폴리오 상태(손익·비중) | 시세·잔고 원천 확보 | 지표 정확도·평가 캐던스 설계, 폴링 cross 누락 |
| 알림 전달 | (인프라) | 설계 단계 | 평가 루프가 멈춰도 사용자는 모른 채 알림을 놓침(조용한 실패) → 헬스체크(liveness 모니터링) 필요 |

**다음 단계로 미루는 것:** 뉴스 해석·주가 영향 예측, 백테스트, 머신러닝, 배당손익, 기간수익률(TWR/MWR), 포트폴리오 스냅샷 비교.

## 1. 기능 A — 포트폴리오 종합 관리

여러 증권사·계좌에 흩어진 보유 자산을 한곳에 모아, 종목·섹터·시장·통화별 비중과 수익/손익을 한눈에 보여준다. 보유 ETF는 구성종목으로 분해해 "실제로 어떤 종목에 얼마나 노출됐는지"까지 계산한다. 핵심 난점은 **데이터 정합성** — 여러 계좌·통화·거래가 섞여도 평단·손익·비중이 정확해야 한다.

### 1.1 현재 보유는 잔고 스냅샷이 진실, 거래내역은 보조
포트폴리오의 **현재 상태(보유수량·평단가·평가금액·예수금)는 증권사 잔고 조회가 유일한 진실의 원천**이다. 거래내역에서 역산하지 않는다 — 액면분할·증자·배당재투자에서 오차가 쌓이고, 거래가 하나라도 누락되면 전체가 틀어진다. 잔고는 평단가까지 직접 제공한다.

거래내역(체결)은 진실의 또 다른 원천이 아니라, **잔고만으로는 알 수 없는 실현손익·매매이력을 채우는 보조 데이터**다. 위탁계좌는 기간 체결내역(3개월 단위 TR)으로 과거 거래까지 조회되지만, 연금·비API 계좌는 제한적이라 부족분은 사용자가 보정한다.

### 1.2 입력 방식
| 방식 | 내용 | 검증 |
|---|---|---|
| 자동 동기화(본인 키) | 한투 잔고·체결내역 / python-kis(국내+해외 통합) | 한투 잔고·위탁 기간 체결내역·python-kis balance() 실측 ✅(§3.1) |
| 수동/CSV | 직접 입력·거래내역 업로드 | 공개 API 없는 증권사(삼성·미래에셋)·연금계좌 보완. **멀티브로커 사용자에겐 사실상 주 입력**이므로 입력 UX를 1차에 제대로 설계 |

### 1.3 관리 항목 — 항목별 원천·계산·검증
손익·비중은 **우리 계산을 표준**으로 두고, 증권사 제공값은 보조·검증용으로 쓴다.

| 항목 | 원천 / 계산 | 검증 |
|---|---|---|
| 보유수량 | 증권사 잔고(`hldg_qty`) / 수동 | 한투 실측 ✅(KRW) |
| 평단가 | 증권사 잔고(`pchs_avg_pric`) / 거래내역 시 이동가중평균 | 한투 실측 ✅(KRW·국내 한정, 해외/USD 경로 미실증) |
| 현재가 | **한투 시세 API**(국내·해외) / yfinance(미국, 미검증) | pykrx는 제외(§3.2) |
| 매입금액 | 계산: 평단 × 수량 | - |
| 평가금액 | 계산: 현재가 × 수량 | ✅ |
| 평가손익(금액/%) | 계산: 평가−매입 / ÷매입. 평가손익률은 일반계좌는 증권사가 제공하나 **연금계좌는 미제공**이라 자체 계산 | ✅ |
| 종목 비중 | 계산: (ETF 분해 반영) 종합 평가금액 ÷ 총평가 | ETF 데이터 의존(§1.5) |
| 예수금 | 증권사 잔고(`dnca_tot_amt`) / 수동 | 한투 실측 ✅ |
| 총매입·총평가·총손익·총자산·현금비중 | 계산: 합산 | - |
| 섹터별 비중 | 계산: ETF 분해 후 종목→섹터 매핑 합산 | 분류 원천 확보(§3.2). 미국·국내 체계 통일 매핑 필요 |
| 시장별·통화별 비중 | 계산: 종목 마스터 메타 | - |
| 원화 환산 | 계산: 통화별 평가금액 × 평가시점 환율 | 환율 §1.7 |

### 1.4 조회 기준(뷰)
| 뷰 | 항목 |
|---|---|
| 요약 대시보드 | 총자산·총평가손익(미실현)·현금비중·자산구성 미니차트 |
| 종목별 보기 | 평단·수량·현재가·평가금액·평가손익(금액/%)·비중 |
| 비중 분석 | 종목별/(조건부 섹터별)/시장별/통화별 비중 |
| 계좌별 보기 | 계좌 집계·비교 |
| 손익 보기 | 미실현 평가손익 + 실현손익(기간: 이번달·지난달·올해·작년·지정) |

- 손익 보기 제약: 실현손익은 해당 기간 거래내역이 있어야 정확(자동 API 조회기간 한계 → 과거는 수동 보정 범위까지).

### 1.5 ETF 분해 (look-through)
보유 ETF를 구성종목으로 풀어 직접 보유분과 합산해, 종목·섹터의 실제 비중을 구한다.
- **계산:** `종목 종합비중 = (직접보유 평가금액 + Σ ETF 평가금액 × ETF내 비중) ÷ 총평가금액`
- **종목 식별자 통일:** 국내는 6자리 종목코드, 미국은 ticker로 체계가 달라 같은 종목을 매칭하려면 공통 키가 필요하다. 국제 표준 식별자(ISIN)로 통일한 종목 마스터를 둔다.
- **비중 합 ≠ 100%:** 현금·파생·반올림으로 구성종목 weight 합이 100%가 아닐 수 있다 → "기타" 버킷으로 분리, 재정규화 정책 고정.
- **중첩 ETF:** ETF가 다른 ETF를 보유하면 재귀 분해(깊이 제한·순환 방지).
- **구성종목 원천:** 미국 SPDR XLSX는 순수 코드로 안정 수집(✅). iShares는 섹터·식별자가 풍부하나 봇차단으로 헤드리스 브라우저가 필요. 국내는 무료·합법·견고한 경로가 없어 미해결(§3.3). → **적용 범위(미국만 vs 국내 포함)는 회의 결정**(§7).
- **섹터 공백:** 구성종목 파일에 섹터가 없을 수 있어(SPDR XLSX 섹터 빈칸), 섹터 비중은 종목→섹터 분류 매핑을 별도로 적용한다. 분류 원천은 미국 yfinance·국내 Naver 업종 등으로 확보(§3.2). ETF는 섹터가 비어 오므로 look-through 후 구성종목에 매핑을 적용한다.

### 1.6 손익 지표 & 정합성 보정
"수익률"은 *(미실현/실현) × (시점/기간)* 조합이라 별도 지표로 분리 제공한다.

| 지표 | 의미 | 데이터 | 1차 |
|---|---|---|---|
| 평가손익(미실현) | 보유분 현재 손익 = "현재 계좌 수익률" | 보유+현재가 | 포함 |
| 실현손익 | 기간 매도 확정 손익 = "올해 수익률" | 거래내역+평단 | 포함(거래내역 있는 범위) |
| 배당손익 / 기간수익률(TWR·MWR) | — | 현금흐름·스냅샷 | 다음 단계 |

**정합성 보정:**
- **기업행위(액면분할·증자·합병):** `CorporateAction{instrument, type, ratio, ex_date}`. 분할이 매수와 매도 사이에 끼면 재구성한 평단이 부호까지 틀어진다(예: 10주@100 → 2:1 분할 → 20주@50인데 거래내역엔 매수 10@100만 남음). ex_date 기준으로 수량·단가를 조정한다. 분할 이력 데이터가 없으면 "실현손익 부정확" 경고를 표시.
- **불완전한 거래내역의 평단 보정:** 거래내역이 종목을 처음 산 시점부터 다 있지 않으면(예: 최근 3개월치만) 평단 계산의 출발점이 틀려, 이후 매도의 실현손익이 전부 어긋난다. 이때는 **잔고 스냅샷의 평단을 출발 평단으로 삼아 보정**하고, "포지션 개시부터 거래내역이 완전할 때만 실현손익 신뢰" 플래그를 둔다.
- **입출금/현금흐름:** 예수금 스냅샷은 입금·매도대금·배당을 구분 못 해 총자산 증감이 수익으로 오인됨 → 최소 `Cashflow{account, type(DEPOSIT/WITHDRAW/DIVIDEND/FEE), amount, currency, at}` 도입, 1차에 부담되면 "총자산 변화 ≠ 수익(입출금 포함)"을 UI 고지.

### 1.7 환율 / 통화
- 기본 원화 통합 환산 + 통화별 축. 환율은 **평가 시점 환율을 함께 기록**.
- **소스:** python-kis `exchange_rate`/`purchase_amount_krw`(필드 확인, 외화 보유 후 실값 검증).
- **매입시점 환율 한계:** 원가 환율(과거)은 자동 확보가 어렵다(python-kis는 현재 환율 위주). → **1차 정의 고정: 원화 평가손익은 "가격손익 + 환손익 혼재값"**으로 정의(둘을 분리하지 않음). `purchase_amount_krw`가 매입 시점 환율인지 현재 환율인지는 추가 검증 필요(§7.2).

```

        ▼
[수집·동기화]  한투 API / python-kis  →  원천 데이터 적재
        ▼
[정합성 계산 레이어]
   · 평단/실현손익(이동가중평균 + CorporateAction 보정 + seed 보정)
   · ETF 분해 합산(미국) · 통화 환산(평가시점 FX 기록)
   · as-of 기준시각으로 정렬
        ▼
[집계]  종목별 / (조건부 섹터) / 시장 / 통화 / 계좌 비중·손익
        ▼
[5개 뷰 서빙]  요약 · 종목별 · 비중분석 · 계좌별 · 손익
```

## 2. 기능 B — 매수/매도 타이밍 알림

### 2.1 개요
사용자가 "이런 상황이 되면 알려줘"를 미리 걸어두면, 시스템이 시세·지표·포트폴리오 상태를 주기적으로 평가해 충족 시 알림을 보낸다. 주문은 안 한다. 조건은 **미리 만든 조건 템플릿("프리셋")을 고르고 기준값만 바꾸는** 방식이다.

각 프리셋은 내부적으로 **`[지표] [연산자] [기준값]`** 데이터로 표현된다(예: `RSI less_than 30`, `5일선 cross_above 20일선`, `평가손익률 less_equal -10%`). 연산자는 7개: cross_above, cross_below, greater_than, less_than, greater_equal, less_equal, equals. 규칙이 데이터라 새 조건 추가 = 데이터 한 줄.

### 2.2 설계 원칙
1. 매수=기회 탐지(가격·지표) / 매도=리스크 관리(손절·익절·트레일링·리밸런싱).
2. 규칙은 선언형 데이터.
3. 사람이 최종 결정(자동 주문 없음).
4. 일 단위 지표 + 시세 폴링으로 판정(장기·스윙 대상).

### 2.3 도메인: PresetDefinition + AlertRule + RuleState
- **PresetDefinition**(시드): `{카테고리, 지표, 연산자, 기본 파라미터, 적용가능 scope_type}` — "프리셋이 무엇인가". 1차는 팀이 시드 정의, 운영자 UI는 다음 단계.
- **AlertRule**(사용자): `{user, preset, params, scope_type(SINGLE/WATCHLIST/ALL_HOLDINGS/SECTOR/THEME), scope_ref, 상태}` — scope 폴리모픽. "ALL_HOLDINGS(전 보유)"는 동적 집합.
- **RuleState**(평가 상태): `{rule, instrument, peak(트레일링 고점), last_condition_met, last_value, updated_at}` — 트레일링 스탑·엣지 트리거 판정에 필수.
- **지표↔스코프 호환 강제:** 평가손익률(손절) 프리셋은 보유 종목에만 — 관심종목(미보유)엔 적용 불가하도록 PresetDefinition.적용가능 scope로 검증.

### 2.4 트리거 정의 UX — 가이드형
빈 캔버스를 주지 않는다. **목적 → 프리셋 → (범위+기준값)** 3스텝.
- 목적: 저가 매수 / 손절·리스크 / 익절 / 리밸런싱
- 프리셋 선택 → 적용 범위(이 종목/관심종목/전 보유/섹터) + 기준값(**추천 기본값** 제공)
- 자유 조합 빌더는 1차에서 숨김.

예상 시나리오: ①"삼성전자 6만원 오면 추가매수"(목표가) ②"보유 −10%면"(손절) ③"반도체 비중 40% 초과"(섹터 과편중) ④"관심종목 RSI<30"(과매도 매수) ⑤"엔비디아 고점 −15%"(트레일링)

### 2.5 프리셋 카탈로그
| 구분 | 1차 (일봉 지표 + 폴링 + 포트폴리오 상태) | 다음 단계 |
|---|---|---|
| 매수 | 목표가/평단 대비 · 52주 최저 근접 · 골든크로스 · RSI 과매도 · 이격도 · N일 연속 하락 · 거래량/변동성(일봉) | 실시간 급변 · 펀더멘털 스크린 · 뉴스 이벤트 |
| 매도 | 손절(−X%) · 익절(+X%) · 트레일링 스탑 · 데드크로스 · RSI 과매수 · 비중 이탈 · 섹터 과편중 | 돌파 실패 · 실시간 급변 |

### 2.6 알림 생명주기 & 전달 (프로세스 + 정합성)
```
[AlertRule 등록] →(룰 생성 시 기존충족 정책 적용)
   │
   ▼  [평가 스케줄]  EOD 일배치 지표 1회 + 장중 가격 폴링(별도 캐던스)
   ▼  [조건 평가]  RuleState 비교
        · 엣지 트리거: last_condition_met=false → true 일 때만 발화
        · 히스테리시스 밴드: 임계 근처 플래핑 차단(예: RSI>35 회복 후 재무장)
        · 쿨다운: 종목·규칙별 시간 제한
   ▼  [Signal 생성] → [Notification 생성]
   ▼  [전달]  인앱 피드(영속) + 푸시(WebSocket/SSE), 오프라인 시 푸시/이메일 폴백
        · at-least-once + 수신측 멱등(notification_id) + 읽음 상태
```
- **폴링 한계(정확히 구분):** 일봉 cross(골든크로스 등)는 *마감된 일봉*에서 결정되어 누락 없음. 누락은 *장중 목표가 터치*에만 해당 → 빠른 장중 움직임은 실시간(§2.7)이 필요함을 명시.
- **룰 생성 시 기존충족 정책:** 등록 시 이미 조건이 참이면 즉시 발화할지, 무음 무장(다음 엣지부터)할지 — 1차 정책으로 고정(권장: 무음 무장).
- **signal_strength:** 1차 단일조건이라 미사용 → **필드 보류**(복합조건 도입 시 추가).

### 2.7 실시간 트리거 — 검증 완료, 1차 편입은 회의 결정
체결틱 기반으로 "장중 눌림목 매수·이상 급변 경보" 트리거가 가능함이 실측으로 확인됐다.
- **국내 틱 수신 실측 ✅:** raw(60초/3종목 707틱), python-kis(15초 27틱) 라이브 수신. **급락·거래량 급증 산출 실증**(예: 035720 +9.35% 포착), 체감 지연 1초 미만.
- **집계 방식:** 분봉 실시간 TR이 없어 **체결틱 → 슬라이딩 N초/1분 버킷 자체 집계**(라이브로 동작 실증).
- **데이터:** 국내 체결 `H0STCNT0`/호가 `H0STASP0`. 미국 `HDFSCNT0` 0분 지연 무료(틱은 정규장 검증 예정, 메커니즘 동일).
- **종목 수:** 위탁계좌(01) appkey로 **41종목 실측 ✅**(IRP는 3종목 예외). 체결+호가 동시 구독이면 종목당 2건이라 ~20종목, 30~50종목+호가는 다중 appkey 샤딩.
- **1차 판정:** 위탁계좌 보유·검증으로 기술 전제는 확보. 1차 범위 포함 여부는 회의 결정(§7).
- **구현 분담:** 잔고는 python-kis, 실시간은 raw WebSocket(python-kis는 캡 초과 조용히 실패, 구독당 REST 추가호출, Decimal 지수표기·domestic/foreign 오분류 → currency 사용 주의).

### 2.8 학습 가이드 (B 진행 전 보강 필요)
알림 엔진은 추가 학습이 필요한 영역이다. 착수 전 다음을 학습·정리한다.
- **기술적 지표 계산:** 이동평균·골든/데드크로스·RSI·볼린저밴드·이격도·거래량 지표 (일봉 OHLCV로 자체 계산)
- **트리거 이론:** 레벨 트리거 vs 엣지 트리거, 히스테리시스(플래핑 방지), 디바운스/쿨다운
- **멱등 메시징:** at-least-once 전달 + 수신측 멱등키, dedup, 재전송
- **실시간 서빙:** WebSocket vs SSE, 백프레셔, 재연결
- **슬라이딩 윈도우 집계:** 체결틱 → N초/1분 버킷(급락·거래량 급증 산출, §2.7)

## 3. 데이터 / API 기술 검토

**원천 용어 간단 설명**
- **한투 OpenAPI / python-kis:** 한국투자증권 공식 API와, 그것을 감싸 국내+해외 잔고를 한 번에 주는 파이썬 래퍼.
- **SPDR XLSX / iShares CSV:** 미국 ETF 운용사(State Street / BlackRock)가 공개하는 구성종목 파일. SPDR은 정적 URL로 바로 받히고, iShares는 봇차단으로 헤드리스 브라우저가 필요.
- **KRX / pykrx:** 한국거래소 데이터와 그것을 스크래핑하는 파이썬 라이브러리. 국내 ETF 구성종목·시세 원천이나 봇차단·약관 제약.
- **krsec:** 여러 국내 증권사(한투·키움·LS·토스)를 한 인터페이스로 묶는 오픈소스 게이트웨이(신생).
- **yfinance / Alpha Vantage / FMP:** 미국 시세·ETF 데이터를 주는 무료·상용 데이터 소스(폴백 후보).

### 3.1 한투 API 실측 (2026-06-28~29, IRP 계좌 기준)
- **인증:** 토큰 발급·캐시 재사용 정상, 유효 1일.
- **잔고:** 보유수량·평단·평가금액·평가손익·예수금 전부 수신. **단 전부 IRP(29)/KRW·USD 미보유 환경** → 평단·환율·원화환산의 **해외/USD 경로는 미실증**. 연금은 평가손익률 미제공(자체계산).
- **실시간 틱:** 국내 라이브 수신·급락/거래량 급증 산출 **실측 ✅**, 지연 1초 미만. 미국 틱은 메커니즘 동일·검증 예정.
- **계좌 유형:** IRP(29)=조회·시세만, 실시간 **3종목 캡**, 매수가능·기간 체결내역 TR 차단(APBK1744). 위탁(01)=전부(주문·실시간·체결내역), DC(55)=API 불가. (사용자는 IRP·위탁 둘 다 보유 → 본운용은 위탁 키로 IRP 제약을 우회)
- **실시간 종목 한도:** **위탁(01)·모의 appkey 모두 41종목 실측**(41 초과 시 OPSP0008), IRP(29)만 3종목 예외. (41은 체결가만 구독 기준 — 체결+호가 동시면 종목당 2건이라 ~20종목)
- **기간 체결내역:** **위탁계좌 실측 ✅** — TTTC0081R(3개월내)·CTSC9215R(그 이전)로 과거 구간 조회, output1 36필드(체결일자·종목·매수매도·수량·체결평균가·추정제비용 등)·output2 요약. IRP는 차단(APBK1744).
- **python-kis:** `balance()` 국내+해외+환율 통합, raw와 수치 일치(실측 ✅).
- **호출 제한:** 실전 초당 ~20건(앱키 단위).

### 3.2 원천 매트릭스
| 용도 | 1차 원천 | 비고 |
|---|---|---|
| 보유/잔고·예수금 | 한투 잔고 / python-kis | 비API·연금은 수동 |
| 거래내역 | 한투 위탁계좌 기간 체결내역(실측✅) / 수동·CSV | 연금·비API는 수동 |
| 현재가·일봉 OHLCV | **한투 시세 API**(국내·해외) / yfinance(미국, **미검증**) | pykrx 제외(JS차단+약관) |
| 실시간 시세 | 한투 WebSocket(국내 실측✅, 미국 0분 무료) | 위탁계좌 41종목 실측(§2.7) |
| ETF 구성종목 | 미국 SPDR(주)·iShares 헤드리스(보조) | 국내 디퍼·벤더 폴백 |
| 섹터 분류 | 미국 yfinance(Yahoo 11섹터)·SEC SIC(공공도메인) / 국내 Naver 업종(WICS, 순수 curl)·data.go.kr(공공) | GICS 라이선스 회피 가능, 체계 통일 매핑 필요 |
| 환율 | python-kis `exchange_rate` | 외화 보유 후 실값 검증 |

### 3.3 수집 환경 제약 — 일부 스크래핑 원천의 봇차단

이 제약은 스크래핑이 필요한 일부 원천에만 해당한다 — **국내 ETF 구성종목(KRX 계열)과 미국 iShares 경로**가 막히고, 인증 기반 공식 API(한투 잔고·시세·실시간)와 정적 파일(미국 SPDR)은 영향이 없다. (국내 섹터 분류도 KRX 직접은 막히나 Naver·yfinance로 우회 가능 — §3.2.)

원인은 IP 종류와 무관한 **JavaScript 기반 봇 차단**(KRX=agent.js 게이트, iShares=Akamai)이다. 브라우저(JS 실행)는 통과하지만 순수 코드 수집(requests·curl)은 차단된다.

| 데이터 | 순수 코드 수집 | 비고 |
|---|---|---|
| 한투·토스 OpenAPI (잔고·시세·실시간) | 🟢 가능 | 인증 API |
| 미국 SPDR XLSX | 🟢 가능 | 정적 URL |
| 국내 ETF 구성종목 (KRX/pykrx) | 🔴 불가 | JS 게이트 + KRX 약관상 자동호출 금지 (섹터는 Naver/yfinance로 우회) |
| 미국 iShares CSV | 🔴 불가 | Akamai → 헤드리스 브라우저 필요 |

- 섹터 공백 같은 "데이터 모양" 문제는 종목 마스터 조인으로 해소 가능하나, 그 **분류 원천 자체가 미확보**다. 봇차단·약관·재배포 같은 접근·법적 제약은 회피/완화만 가능하다.
- 완화책: 헤드리스 브라우저 자동화(UI 변경에 취약), 정적 파일 우선, 유료 벤더 API 폴백, 운용사 파일 파싱.

### 3.4 기술 검토 결론 (요약)
- **검증 완료(✅):** 한투 잔고(KRW)·python-kis 통합 잔고·국내 실시간 틱/급락·거래량 산출·실시간 41종목 한도(위탁·모의 실측)·위탁 기간 체결내역·SPDR XLSX 수집·환율 필드 존재·섹터 분류 원천(미국 yfinance, 국내 Naver 업종).
- **미해결/미실증(⚠️):** 해외/USD 잔고·평단·환율 실값, 미국 실시간 틱, 섹터 체계 통일 매핑, 국내 ETF 구성종목 합법 경로, yfinance·토스 시세(가격), 매입시점 FX.
- **차단 확정(🔴):** KRX/iShares 순수 코드 수집(JS 봇차단), pykrx(약관), IRP 실시간 3종목 캡, 멀티유저+개인 API custody.
- **결론:** 1차 핵심(포트폴리오 잔고 자동화 + 일봉 지표 알림)은 **한투 API + python-kis로 본인 계좌 범위에서 충분히 구현 가능**. 실시간·국내 ETF·섹터·멀티유저는 추가 전제(위탁계좌·원천·아키텍처)가 필요해 디퍼/조건부.

## 4. 도메인 모델 (임시 — 7/7 확정)

### 4.1 엔티티
| 엔티티 | 핵심 속성 | 역할 |
|---|---|---|
| `User` | id | 사용자(=본인) |
| `Account` | 증권사, 계좌유형(일반/연금), 기준통화, 원천, last_synced | 계좌 |
| `Instrument` | ISIN, 종목코드/ticker, 이름, type(STOCK·ETF), market, currency, sector | 종목 마스터 |
| `ETFConstituent` | etf, underlying, weight, as_of | ETF 구성(자기참조) |
| `Holding` | account, instrument, 수량, 평단, snapshot_at | 현재 보유(스냅샷) |
| `Transaction` | account, instrument, type, 수량, 단가, 일시, 비용 | 거래내역 |
| `CorporateAction` | instrument, type, ratio, ex_date | 분할·증자·합병 |
| `Cashflow` | account, type(입금·출금·배당·수수료), amount, currency, at | 현금흐름 |
| `Watchlist` | user, instrument | 관심종목 |
| `Sector`/`Theme` | 분류 (Theme↔Instrument N:M) | 섹터·테마 |
| `PresetDefinition` | 카테고리, 지표, 연산자, 기본 파라미터, 적용가능 scope_type | 알림 프리셋 |
| `AlertRule` | user, preset, params, scope_type, scope_ref, 상태 | 사용자 규칙 |
| `RuleState` | rule, instrument, peak, last_condition_met, last_value, updated_at | 평가 상태(트레일링·엣지) |
| `Signal` | rule, instrument, 발생시각, 평가 스냅샷 | 충족 이벤트 |
| `Notification` | signal, user, 채널, 상태(생성·전달·읽음), notification_id | 알림 전달 |
| `FxRate` | currency_pair, rate, as_of | 환율(시점) |

> `Portfolio`는 테이블이 아니라 Holding·Transaction 집계 파생.

### 4.2 관계
```
User ─< Account ─< Holding   >─ Instrument ─< ETFConstituent >─ Instrument
              ├─< Transaction >─ Instrument ├─ Sector
              └─< Cashflow                  └─< CorporateAction
User ─< Watchlist >─ Instrument            Instrument >─< Theme
User ─< AlertRule >─ PresetDefinition
            │  └ scope_type+scope_ref → Instrument/Watchlist/ALL_HOLDINGS/Sector/Theme
            └─< RuleState,  AlertRule ─< Signal ─< Notification
FxRate(currency_pair, as_of)
```

### 4.3 핵심 불변식
- 현재 보유의 진실은 `Holding`(스냅샷). 역산 금지.
- 평단: 거래내역 기반 이동가중평균 + **CorporateAction(ex_date) 보정** + 부분내역 시 스냅샷 평단 seed. 매도 시 `실현손익=(매도가−평단)×수량−비용`.
- look-through 합산: weight 합 ≠ 100%는 "기타" 버킷, 중첩 재귀(깊이 제한).
- **환율 정합:** 원화 환산값은 적용 `FxRate.as_of`와 짝으로 **머티리얼라이즈해 기록**한다(매번 런타임 계산만 하면 "어느 시점 환율로 환산했는지" 기록이 남지 않는다).
- **as-of 기준시각:** 계좌별 snapshot_at·FxRate가 제각각이면 총자산/비중 시점 불일치 → 집계 시 단일 as-of 기준 적용.
- 알림 멱등: `RuleState`(직전 상태) + 엣지 + 히스테리시스 + 쿨다운, `Notification.notification_id` 수신측 멱등.
- 데이터 충돌: 같은 종목 스냅샷 vs 수동 → 스냅샷 우선(수동은 비API 계좌 전용).

## 5. 보안 · 비용 · 운영

- **자격증명 custody(본인 키 모델):** 1차는 본인 키로 본인 계좌만 → 타인 자격증명 보관 없음. 키는 환경변수/암호화 저장, 사용자가 직접 입력. **멀티유저 전환 시 KMS·암호화·유출 시나리오·약관 재검토 필수**(현 모델의 단순함은 멀티유저에서 무효).
- **Rate-limit/비용:** 앱키당 ~20req/s. 본인 키 모델이라 사용자별 독립. 멀티유저면 사용자당 한도에 묶이고 fan-out 설계 필요(디퍼).
- **평가기 liveness 모니터링:** 평가 루프가 죽으면 알림이 조용히 끊김(신뢰 기반 제품의 치명적 침묵 실패) → 헬스체크·알람 필수(데이터팀 운영).
- **개인정보·면책:** 포트폴리오·자격증명은 민감정보 → 접근통제·암호화. "투자자문 아님·손해 면책" 고지. 타인 자산 대리 취합은 마이데이터 규제 영역이라 본인 키 모델로 회피.

## 6. 협업 경계 · 팀 과제 · 리스크

### 6.1 협업 경계
- **백엔드:** 포트폴리오 도메인·정합성 계산·look-through·알림 규칙 평가·서빙.
- **데이터팀:** 메달리온 + 람다 데이터 파이프라인 구축 — 수집·처리·서빙(상세 6.3).
- **인터페이스(공백 보강):**
  - **ISIN 종목 마스터 소유권** → 데이터팀이 소유, 백엔드는 조인만.
  - **라이브 시세 서빙 경계** → 목표가 알림용 근실시간 현재가를 데이터팀 서빙 테이블에서 받을지/백엔드가 한투 직접 폴링할지 명시(권장: 데이터팀 서빙 테이블 + 프레시니스 SLA).
  - **거래내역 보정 머지**(한투 부분 + CSV dedup·평단 seed) → 백엔드 책임.

### 6.2 백엔드 과제 (연산·정합성)

**포트폴리오 계산**
- **평단가 재구성:** 거래내역으로 이동가중평균 평단 계산. 분할 매수/매도 시 평단 갱신, 매도 시 `실현손익 = (매도가 − 평단) × 수량 − 비용`. 거래내역이 불완전하면 잔고 스냅샷 평단을 출발점으로 보정.
- **기업행위 보정:** 액면분할·증자·합병(CorporateAction)을 ex_date 기준으로 수량·단가에 반영해 평단·실현손익이 틀어지지 않게 함.
- **ETF look-through 합산:** 직접보유 + Σ(ETF 평가 × 구성종목 비중)을 ISIN 기준으로 합산. 중첩 ETF 재귀 분해, weight 합 ≠ 100% 시 "기타" 버킷·재정규화.
- **통화·환율:** 통화별 평가금액을 평가 시점 환율로 원화 환산하고, 적용 환율 시점을 함께 저장(머티리얼라이즈)해 정합 유지.
- **as-of 정렬:** 계좌마다 다른 스냅샷 시각·환율 시점을 하나의 기준시각으로 맞춰 총자산·비중의 시점 일관성 확보.
- **데이터 충돌 해결:** 같은 종목이 자동 잔고와 수동 입력에 동시 존재할 때 우선순위(스냅샷 우선), 한투 부분 거래내역 + 사용자 CSV의 중복 제거·평단 seed 머지.

**알림 엔진**
- **규칙 평가:** 선언형 규칙(`지표 연산자 기준값`)을 해석해 종목/관심종목/전 보유/섹터 스코프에 적용. 지표↔스코프 호환 검증(손절은 보유 종목에만).
- **지표 계산:** 일봉 OHLCV로 이동평균·RSI·볼린저·이격도·거래량 등 자체 계산(장 마감 후 배치).
- **트리거 상태관리:** 엣지 트리거(직전 충족 상태 추적), 히스테리시스(플래핑 방지), 쿨다운, 트레일링 스탑의 고점(high-water-mark) 유지 — RuleState로 관리.
- **알림 전달:** Signal→Notification 생성, at-least-once 전달 + 수신측 멱등(notification_id), 인앱 피드 영속화 + 실시간 푸시(WebSocket/SSE), 읽음 상태.

**API·서비스**
- 도메인 API 설계(포트폴리오 5개 뷰 조회, 관심종목·알림 규칙 CRUD), 트랜잭션 경계 설정, 동기화 작업 오케스트레이션.

### 6.3 데이터팀 과제 (데이터 파이프라인)
데이터팀은 메달리온 + 람다 구조의 데이터 파이프라인을 구축한다.
- **수집:** 잔고·거래·시세·환율·ETF 구성종목(한투/python-kis/SPDR/iShares 헤드리스), 호출 제한·캐싱.
- **메달리온(Bronze→Silver→Gold):** Iceberg에 원본 적재 → 정규화·중복제거 → 지표·피처 테이블. Spark 배치로 일봉 지표(이동평균·RSI 등)와 종목 마스터(ISIN·섹터)를 계산.
- **람다 스피드 레이어:** 실시간 체결틱을 스트림 처리해 급락·거래량 급증을 산출(위탁계좌 확보 시 활성화).
- **서빙:** 백엔드가 소비할 라이브 시세 서빙 테이블 + 프레시니스 SLA, Gold 피처 테이블.
- **운영:** 파이프라인·평가기 liveness 모니터링.
- (다음 단계) 뉴스 이벤트 파이프라인.

### 6.4 예상 리스크
- 멀티유저 전환 시 한투 개인 API custody·약관(현 1차는 본인 키로 회피)
- 섹터 체계 불일치(미국 Yahoo 11 vs 국내 WICS) → 공통 매핑 필요, 스크래핑 원천은 개인용 한정
- 국내 ETF 구성종목 합법·견고 경로 부재
- JS 봇차단(KRX·iShares) 헤드리스 의존(취약), KRX 약관(상업화 시 발효)
- 평가기 침묵 실패, 매입시점 FX 부재(원화 손익에 환손익 혼재)
- krsec 미성숙, 무료 데이터 한도

## 7. 열린 항목 — 회의 결정 · 검증 대기

### 7.1 회의에서 결정할 것 (팀 결정)
1. **1차 범위 — 실시간·데이터팀 산출물:** 메달리온 배치 파이프라인(기본) 외에, 실시간 스피드레이어(위탁계좌로 41종목·체결내역 실측 완료, 기술 전제 확보)나 뉴스 파이프라인을 1차에 넣을지. 셋의 역량 균형 관점에서 결정.
2. **ETF 분해 적용 범위:** 미국만 vs 국내 포함. 국내는 무료·합법·견고한 구성종목 원천이 없어, 헤드리스 수집·운용사 파싱·유료 벤더 중 무엇을 감수할지와 함께 결정.
3. **멀티유저/상업화 시점:** 1차는 본인 키 모델 확정. 상업화 검토 시 입력 모델(수동 주경로)·custody·약관 재설계.

### 7.2 검증·조사 대기 (외부 의존)
4. **국내 ETF 구성종목 수집 원천** — 헤드리스 / 운용사 파싱 / 벤더.
5. **매입시점 FX** — python-kis `purchase_amount_krw`의 환율 기준(매입시점 vs 현재) 검증.
6. **미국 실시간 틱·USD 환율 실값** — 미국 정규장·외화 보유 시 검증.
